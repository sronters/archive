Source: `C:\Users\User\Downloads\Telegram Desktop\ТЗ_Кейс2_MedArchive.docx`.

## Цель

Создать систему автоматической обработки архива прайс-листов клиник-партнеров, которая:

- разбирает входящие документы разных форматов: PDF, DOCX, XLSX/XLS, сканы;
- извлекает структурированные данные: клиника, услуга, цена, тариф резидент/нерезидент;
- нормализует данные по единому целевому справочнику услуг;
- формирует верифицированную базу партнеров, услуг и цен;
- предоставляет API и интерфейс поиска: кто оказывает услугу и по какой цене.

Целевой справочник медицинских услуг предоставляется организаторами в XLSX или JSON. Система должна предусмотреть механизм загрузки справочника и привязки извлеченных услуг к нему.

## Входные данные

Архив ZIP содержит прайс-листы клиник-партнеров:

- PDF из DOCX: структурированный текст, таблицы читаются корректно;
- PDF-скан: изображение документа, требуется OCR;
- XLSX/XLS: табличные данные, возможны множественные листы;
- DOCX: текстовые документы с таблицами.

Один файл обычно соответствует прайсу одной клиники-партнера на конкретную дату. Имя файла может содержать название клиники и дату. Для одного партнера может быть несколько прайсов, все доступные прайсы должны учитываться для истории изменения цен.

Минимальный состав справочника услуг:

- `service_id`: UUID услуги;
- `service_name`: официальное название;
- `synonyms`: список синонимов и альтернативных названий;
- `category`: категория услуги.

## Базовые Сущности Из ТЗ

`Partner`:

- `service_name_raw`, `service_code_source`;

- `price_original`, `currency_original`;

`Service`:

- `service_id`, `service_name`, `synonyms`, `category`;
- `icd_code`, `is_active`.

Также нужны пользовательские и операционные сущности: `User` или `Operator`, `VerificationTask`, `PriceHistory`.

## Функциональные Требования

Загрузка и первичная обработка:

- прием ZIP через UI или CLI;
- автоматическое определение типа каждого файла;
- постановка файлов в очередь обработки со статусом;
- сохранение оригинальных файлов для повторной обработки.

Извлечение:

- текстовый PDF: извлечение текста и таблиц через pdfplumber, PyMuPDF или аналог;
- PDF-скан: OCR через Tesseract или облачный OCR при разрешении;
- XLSX/XLS: обход всех листов, определение header row, извлечение колонок услуг и цен;
- DOCX: работа с финальной версией текста, включая tracked changes, и извлечение таблиц.

Нормализация и сопоставление:

- сопоставление `service_name_raw` со справочником по точному совпадению, синонимам и нечеткому поиску;
- порог уверенности должен быть конфигурируемым;
- несопоставленные позиции попадают в очередь `unmatched`;
- оператор выбирает услугу из справочника или корректирует результат.

Валидация:

- цена должна быть числом и больше нуля;
- цена нерезидента не должна быть меньше цены резидента;
- название услуги не должно быть пустым;
- дата прайса не должна быть в будущем;
- дубли по клинике, услуге и дате должны дедуплицироваться;
- изменение цены больше 50% должно отмечаться как аномалия;
- валюта не KZT должна конвертироваться по курсу на дату прайса с сохранением оригинала;
- документ без распознаваемых данных получает статус ошибки.

Ручная верификация:

- оператор видит исходный документ или фрагмент рядом с извлеченными данными;
- оператор подтверждает, отклоняет или корректирует позицию;
- при изменении цены старая версия архивируется и не удаляется.

## API И UI

REST API должен иметь OpenAPI/Swagger документацию.

Базовые endpoints из ТЗ:

- `GET /services` - список услуг справочника с фильтрацией;
- `GET /services/{id}/partners` - партнеры, оказывающие услугу, с ценами;
- `GET /partners` - партнеры с фильтрацией по городу и статусу;
- `GET /partners/{id}/services` - услуги конкретного партнера с ценами;
- `GET /search?q=` - полнотекстовый поиск по услугам и партнерам;
- `GET /unmatched` - несопоставленные позиции для операторов;
- `POST /match` - ручное сопоставление позиции с услугой справочника.

Пользовательский интерфейс:

- поиск услуги с выводом партнеров и цен резидент/нерезидент;
- страница партнера с полным прайсом, контактами и датой актуальности;
- административная загрузка архива;
- статус обработки;
- очередь верификации;
- дашборд с количеством обработанных документов, процентом нормализации и очередью задач.

## Нефункциональные И Оценочные Требования

Исходное ТЗ задает ориентиры:

- текстовый PDF/DOCX/XLSX: обработка одного документа до 60 секунд;
- скан с OCR: до 3 минут на документ;
- исходные файлы и сырые данные не удаляются;
- история изменений цен хранится бессрочно;
- новый источник или формат должен добавляться без изменения ядра.

Критерии оценки:

- качество извлечения данных;
- нормализация и сопоставление;
- валидация, верификация и версионирование;
- API и поиск;
- UX административного раздела.

Важно: формулировка исходного ТЗ про "MVP" и 70% автоматической нормализации не является достаточной планкой для этого репозитория. Для дальнейшей работы применяется production-планка из `quality-bar.md`.
